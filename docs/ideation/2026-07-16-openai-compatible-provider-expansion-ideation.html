What the repository and current docs establish

Grounding Context

BYOK Runtime’s reusable compatibility floor is narrow and useful: send one text prompt through OpenAI-style chat completions and populate a portable model picker. The provider matrix should be evaluated against those operations independently.

Minimum contract: generation calls chat.completions.create({ model, messages }) and extracts assistant text; discovery yields model entries with a nonblank id, normalized to { id, label }. Bearer auth and {baseURL}/models are defaults, not universal truths.

1

Compatibility-led 15-provider expansion program

Axis · Current contractConfidence · 88%Complexity · Medium

Description: Make the 13-provider compatibility ledger the front door, then pursue 12 documented same-family candidates and three catalog-adapter candidates. Label direct vendors, gateways, local servers, and CLI adapters explicitly; documentation earns candidacy, while live admission certificates earn support.

Basis
direct: src/provider-manifest.ts, src/providers/openai-compatible-provider.ts, and the README establish the current 13-provider families and the two-operation subset. external: official docs support the 15-provider candidate portfolio shown above, with catalog caveats preserved.
Rationale
This meets the requested provider breadth without treating every “OpenAI-compatible” claim as identical. It also gives future provider research a reusable decision system instead of another one-off list.
Downsides
The portfolio is documentation-qualified, not credentialed proof. Some candidates may move lanes or drop out after smoke testing.
2

Executable list-then-generate admission certificates

Axis · VerificationConfidence · 94%Complexity · Medium

Description: Require each candidate to authenticate, list models, select a documented chat-capable ID, and generate plain text using that exact result. Store endpoint, catalog shape, tested model, verification date, and outcome as a compatibility receipt; structured-object behavior is a separate capability, not the admission floor.

Basis
direct: BYOK exposes separate models and generate smoke commands, while testConnection() currently requires JSON-object generation beyond the requested text floor.
Rationale
The sequence catches both false positives: a catalog with no usable chat model and a chat-compatible API with a broken model picker. Dated receipts make later re-verification targeted.
Downsides
Live testing requires credentials, billable calls, model-selection rules, and maintenance when catalogs deprecate IDs.
3

One bounded model-catalog strategy seam

Axis · Adapter-requiredConfidence · 92%Complexity · Medium

Description: Preserve OpenAI-client generation, but let the manifest select a narrow model-list strategy: URL or path, required headers/account parameters, response shape, optional pagination, and normalization. This unlocks Fireworks, GitHub Models, and Cloudflare without creating three provider transports.

Basis
direct: the internal provider config already exposes listModelsImpl, requestHeaders, and normalizeModel; the public manifest exposes only one base URL and two small enums. external: the three candidate catalogs diverge exactly at this operation.
Rationale
Model discovery is the common incompatibility. Spending one abstraction there captures the value without generalizing the entire request stack.
Downsides
Account-scoped URLs may require new provider configuration fields. Pagination and filtering can turn a “small strategy” into a mini framework if not tightly bounded.
4

Make the manifest the provider expansion engine

Axis · RolloutConfidence · 91%Complexity · Medium

Description: Before multiplying providers, derive or mechanically validate discovery order, credential maps, public inventories, smoke-test matrices, and provider documentation from the manifest. Keep repeated lists only when they are intentional public contracts.

Basis
direct: the registry and credential maps already derive from the manifest, but provider-discovery.ts, exact-array tests, fixtures, README/API tables, and release metadata still repeat the provider inventory.
Rationale
Without this, adding 15 providers is mostly synchronized bookkeeping. A stronger manifest makes the first expansion compound into cheaper, safer future additions.
Downsides
Not every artifact should be generated, and exact public ordering still deserves explicit tests. Over-generation can make docs harder to curate.
5

Evidence tiers and a dated verification queue

Axis · VerificationConfidence · 93%Complexity · Low

Description: Give each candidate one evidence state: live-verified, documented-awaiting-key, adapter-specified, or held. Keep Z.AI, Hyperbolic, Moonshot/Kimi, Perplexity, Azure OpenAI, DashScope, and Chutes visible with explicit promotion conditions instead of padding the main list.

Basis
external: current official research confirms generation for several held vendors but could not establish a stable, BYOK-usable model-list contract. reasoned: documentation and live conformance are different evidence strengths.
Rationale
This makes uncertainty actionable and lets the catalog evolve without silently weakening the support promise.
Downsides
Evidence states need an owner or cadence, or the queue will become stale documentation.
6

Roll out by failure domain, not provider popularity

Axis · RolloutConfidence · 89%Complexity · Low

Description: Use three independent release trains: zero-runtime-change manifest additions, catalog-adapter providers, then unresolved auth/endpoint/catalog cases. Each train completes compatibility receipts, provider inventories, discovery/tests, smoke coverage, docs, changeset, and the full package check before the next begins.

Basis
direct: provider additions currently cross types, manifest, discovery, exact fixtures, docs, smoke tests, and changeset/release checks. These cohorts fail in materially different places.
Rationale
A failed adapter no longer blocks the direct cohort, and review stays focused on one compatibility pattern at a time.
Downsides
Multiple releases create coordination overhead, and provider ordering must be planned before the stack starts.
7

Generic custom OpenAI endpoint for self-hosted servers

Axis · Drop-in expansionConfidence · 78%Complexity · Medium

Description: After curated presets, consider one caller-configurable base URL, credential, and model-list contract that covers vLLM, llama.cpp, LocalAI, SGLang, and LiteLLM Proxy. LM Studio already proves the configurable-local-URL shape; named engine presets, if ever needed, can remain host presentation rather than new transports.

Basis
direct: LM Studio uses the same OpenAI-compatible provider with a configurable URL. external: the named servers document OpenAI-compatible chat and models endpoints. reasoned: one endpoint escape hatch absorbs future compatible servers more efficiently than an enum entry per engine.
Rationale
This is the highest-leverage way to reach the upper end of the requested 12–24 range without turning every deployment engine into a permanent vendor preset.
Downsides
Custom URLs expand SSRF and trust-boundary concerns; the current library deliberately keeps cloud base URLs fixed. This direction needs a separate security and product-contract discussion.
