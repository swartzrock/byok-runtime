AIChat provider parity for BYOK Runtime

How to reach AIChat's full provider surface without turning a compact TypeScript library into 25 bespoke integrations—or freezing an API that fails at Vertex and Bedrock.

Grounding Context

Target library. byok-runtime is a strict TypeScript/ESM npm library with a deliberately small trusted-host API. It currently recognizes nine provider IDs: five API-key clouds, two local servers, and two local CLIs. Its public contract centers on text generation, optional object generation, model listing, and connection status.

Comparison source. AIChat's source registers seven dedicated transports—OpenAI, Gemini, Claude, Cohere, Azure OpenAI, Vertex AI, and Bedrock—plus one OpenAI-compatible transport carrying 18 named aliases: AI21, Cloudflare, DeepInfra, DeepSeek, Ernie, GitHub Models, Groq, Hunyuan, MiniMax, Mistral, Moonshot, OpenRouter, Perplexity, Qianwen, xAI, ZhipuAI, Jina, and Voyage AI. Ollama is a configurable compatible endpoint rather than a registered alias.

Difficulty signal. Most missing names are catalog records over an existing transport. Native protocol translation is the middle tier. Azure changes routing and headers; Vertex adds renewable OAuth plus project/location/publisher routing; Bedrock adds multi-part AWS credentials, regional endpoints, SigV4 signing, and provider-neutral conversation mapping.

Capability gap. AIChat also models streaming, tools, vision, embeddings, reranking, usage, and detailed per-model constraints. Provider-name parity and operation parity are different milestones and should be reported separately.

1 · Foundation

Define parity by transport family, not provider count

Treat the goal as eight wire families plus named presets, rather than 25 bespoke provider implementations. The working order becomes: compatible presets; Azure's OpenAI-shaped routing; native Claude/Gemini/Cohere codecs; Vertex composition; Bedrock signed transport.

Confidence 96%Complexity LowAxis: compatible presets
Basis
direct: AIChat's src/client/mod.rs places 18 aliases behind one openai-compatible client and registers dedicated clients separately.
Rationale
This taxonomy makes “easier to harder” reflect translation and authentication distance, and prevents one provider file per marketing brand.
Downsides
The word “provider” becomes overloaded; documentation must distinguish identity, preset, transport, and verified capability.
2 · Highest leverage

Build one executable provider manifest

Create a single authoritative specification for provider identity, transport family, default endpoint, environment keys, credential fields, icon/labels, model-list strategy, and bounded quirks. Generate or validate today’s duplicated enum, unions, registry, credentials map, factory cases, documentation rows, and contract tests from it; pin the upstream AIChat source revision and report catalog drift.

Confidence 94%Complexity MediumAxis: compatible presets
Basis
direct: BYOK currently repeats provider identity across types.ts, credentials.ts, registry.ts, icons, factories, and exact-order public-contract tests. AIChat's own ai21/ai12 example drift demonstrates the failure mode.
Rationale
Adding the compatible cohort becomes catalog work with exceptional code only; future upstream additions become detectable rather than another audit.
Downsides
Code generation can obscure public API changes. A validation-first manifest may be safer than immediately generating every source file.
3 · Easy-lane release strategy

Ship compatible providers in confidence rings

Add missing aliases through the shared adapter, but classify each operation as verified, unsupported with reason, or unverified. Start with fixed-URL bearer-auth chat endpoints; follow with templated URLs and optional credentials; finish the compatible lane with operation quirks such as Ernie rerank paths and Voyage field names.

Confidence 92%Complexity MediumAxis: capability contract
Basis
direct: BYOK's existing compatible adapter already parameterizes base URLs, headers, model normalization, and listing. external: AIChat's compatible client exposes optional auth and provider-specific rerank quirks.
Rationale
This provides provider-name breadth early without pretending “OpenAI-compatible” means equal support for listing, objects, embeddings, reranking, or streaming.
Downsides
Confidence rings need crisp promotion tests and could confuse users if labels are too granular.
4 · Semantic parity

Make custom compatible provider instances first-class

Support arbitrary names, base URLs, optional keys, and user-supplied model IDs/capabilities in addition to built-in presets. Separate stable provider kind from configured instance identity so multiple Azure resources, Vertex projects, Bedrock regions, or compatible endpoints can coexist.

Confidence 90%Complexity MediumAxis: compatible presets
Basis
direct: AIChat accepts unknown compatible provider names with custom endpoints and named client configurations; BYOK's current closed enum and single config shape cannot express that class.
Rationale
Static alias parity would still omit a provider class AIChat supports. Instances also model enterprise accounts more honestly than one global enum slot per vendor.
Downsides
This is a larger public-type change than adding enum values and needs a migration story for existing stored settings.
5 · Native-adapter architecture

Use protocol codecs, then compose routing and authentication

Introduce a small provider-neutral representation for messages, content parts, tools, sampling, stream events, embeddings, and rerank inputs. Lower it through OpenAI, Claude, Gemini, and Cohere codecs; keep endpoint routing and request authentication separate so Azure wraps OpenAI semantics and Vertex routes to Gemini, Claude, or Mistral codecs instead of duplicating them.

Confidence 89%Complexity HighAxis: native adapters
Basis
direct: AIChat translates Anthropic content blocks, Gemini roles/parts, and Cohere request/event fields; its Vertex client selects protocol behavior by publisher/model family.
Rationale
The work compounds: direct Claude/Gemini codecs become Vertex prerequisites, Azure reuses OpenAI, and future providers add translations rather than orchestration forks.
Downsides
An over-general IR is a classic abstraction trap. It should grow only from exercised provider fixtures, with Cohere as an early stress test.
6 · Honest capability growth

Publish a model-level capability ledger with optional facets

Keep generateText as the stable minimum, then expose optional streaming, messages/tools/vision, embeddings, and reranking facets. Capabilities belong primarily to model deployments—not just provider brands—and should allow supported, unsupported, and unknown states, backed by static metadata, discovery, and probes.

Confidence 91%Complexity HighAxis: capability contract
Basis
direct: BYOK models currently contain only id and label. AIChat records modality, tools, vision, stream/system restrictions, token limits, pricing, and embedding limits, while providers implement uneven operations.
Rationale
It separates provider-name parity from capability parity and gives consumers typed feature detection instead of prose claims or failed network calls.
Downsides
Capability metadata can stale quickly. Conservative unknown states and monotonicity checks are necessary.
7 · Sequencing rule

Deliver easy-to-hard, but validate architecture hard-to-easy

Ship value in the intuitive order, yet run thin design probes for Vertex OAuth/project routing and Bedrock compound credentials/SigV4 before freezing the compatible-provider config model. The probes need not ship; they only prove that the public credential, endpoint, instance, and capability shapes will survive the final tranche.

Confidence 93%Complexity MediumAxis: cloud identity and routing
Basis
direct: BYOK currently reduces cloud config to {apiKey, model}, while Vertex requires project/location and expiring OAuth and Bedrock requires region, key/secret, optional session token, and per-request signing.
Rationale
It prevents easy aliases from cementing a public API that later needs a breaking rewrite, without delaying the easy-lane release until Bedrock is complete.
Downsides
Architecture probes consume effort before visible providers ship and must be time-boxed to avoid disguised implementation.

Rejection Summary

# Considered idea Reason cut
1 Host-owned credential brokers as a standalone direction Credential abstraction is necessary, but the evidence does not settle ownership; folded into the routing/auth tradeoff.
2 AIChat differential fixture oracle Potentially valuable, but cross-language oracle stability and maintenance cost are not supported strongly enough.
3 Vertex routing shell as a separate idea Strong but narrower duplicate of protocol codecs plus composable routing.
4 Provider lockfile and drift detector Strong but best treated as a property of the executable manifest, not a separate direction.
5 Cohere-first stress test Useful sequencing variant, but it belongs inside native-codec validation rather than the top-level candidate set.
6 Runtime capability negotiation handshake Promising later refinement; remote probes can be unreliable and should not precede explicit static capability states.
7 Capability monotonicity rules Sound maintenance policy, but subordinate to the capability-ledger direction.
8 One native adapter per provider Rejected as duplicative and contrary to direct evidence that most aliases share transports.
