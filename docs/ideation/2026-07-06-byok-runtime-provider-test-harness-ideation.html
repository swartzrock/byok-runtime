BYOK Runtime Provider Test Harness

Grounding Context

byok-runtime is already organized around exactly the two operations the tester needs: text generation and model discovery. The README describes first success as calling generateText with provider, credential, model, and prompt, and setup flows as calling listModels before a model is selected (README.md:12, README.md:14).

The main entrypoint is intentionally the trusted-runtime API for API-key providers and Ollama, while local CLI providers live on @swartzrock/byok-runtime/node (README.md:64, README.md:75). The user's provider table covers Anthropic, OpenAI, Google Gemini, xAI, OpenRouter, and Ollama, so the default harness should stay on the main facade and leave Codex CLI / Claude CLI as a later extension.

The public types keep model discovery portable: ByokModelOption is only id and label, cloud providers use apiKey, and Ollama uses an optional URL plus model (src/types.ts:69, src/types.ts:75, src/types.ts:139). The facade already has separate config constructors for generation and listing, with a sentinel model for listing (src/client.ts:14, src/client.ts:51, src/client.ts:89).

External provider docs support the expected environment convention: Anthropic's TypeScript SDK example uses ANTHROPIC_API_KEY by default; OpenAI's quickstart uses OPENAI_API_KEY and says SDKs auto-read it; Google documents GEMINI_API_KEY or GOOGLE_API_KEY with GOOGLE_API_KEY taking precedence; xAI examples use XAI_API_KEY; OpenRouter examples use OPENROUTER_API_KEY; and Ollama JS defaults its host to http://127.0.0.1:11434.

1Fixture-App CLI Under examples/provider-smoke/

Axis: Packaging and verificationConf 88%Cx Medium

Description: Create a small, private example package under examples/provider-smoke/ with its own package.json, TypeScript CLI entry, and scripts. It should import only the public package entrypoint, accept provider/model/input/key options, and be excluded from package publication unless intentionally added later.

Basis
direct: The package already documents consumer-facing imports from @swartzrock/byok-runtime and treats provider internals as private (README.md:64, README.md:85). package.json publishes only dist, docs, and metadata today (package.json:35).
Rationale
An example app validates the public API exactly as a consumer would use it, while staying out of the library runtime and publish surface. It is also the natural place for real API-key calls that should not run in CI by default.
Downsides
Requires a little workspace wiring and a decision about whether examples typecheck as part of the normal quality gate.

2Two Commands: models and generate

Axis: Command surfaceConf 86%Cx Low

Description: Give the CLI exactly two primary commands. models --provider <id> [--api-key <key>] [--url <url>] lists models. generate --provider <id> --model <model> --input <text> [--api-key <key>] [--url <url>] lists the first five models, then prints the generated response.

Basis
direct: The public facade already exports separate generateText and listModels helpers (src/client.ts:80, src/client.ts:89), and the README separately documents "Fetch Models" and "Generate Text" flows (README.md:189, README.md:202).
Rationale
This mirrors the two product questions a maintainer has while testing providers: can we discover models for this account, and can we get a text response from a chosen model?
Downsides
It does not cover structured output or testConnection() in v1. That is probably correct for the user's immediate request.

3Explicit-Then-Environment Credential Resolver

Axis: Credential and environment resolutionConf 84%Cx Low

Description: Centralize credential resolution in one small map: explicit --api-key wins, then the provider's standard environment variable. Google should check GOOGLE_API_KEY before GEMINI_API_KEY, matching Google's documented precedence. Ollama should ignore API keys and use --url or the BYOK default URL.

Basis
external: Anthropic's TypeScript SDK example names ANTHROPIC_API_KEY as the default key source. OpenAI's quickstart uses OPENAI_API_KEY and says SDKs auto-read the system environment. Google documents GEMINI_API_KEY or GOOGLE_API_KEY, with GOOGLE_API_KEY taking precedence. xAI examples use XAI_API_KEY. OpenRouter examples standardize on OPENROUTER_API_KEY.
Rationale
A visible resolver keeps provider quirks out of the command handlers and gives contributors one place to update when provider conventions change.
Downsides
Environment resolution is host-app behavior, not core library behavior, so it should stay inside the fixture app rather than leaking into src/.

4Model-List-First Generation Output

Axis: Provider behavior coverageConf 82%Cx Low

Description: Make generate call listModels first and print the first five returned model IDs before it calls generateText. This single command proves model discovery and text generation work for the same credential without making the user run two commands during every smoke check.

Basis
direct: The README says setup flows call listModels before a model is selected, and the model options contract is intentionally portable (README.md:14, README.md:200). The facade does not require a selected model for listing (src/client.ts:51).
Rationale
The user asked specifically for response generation plus "also list the first 5 available models." Doing the list first catches bad keys and discovery endpoint regressions before spending a generation call.
Downsides
Some providers may return many models or transient model-list failures even when generation would work. The CLI should report that clearly rather than hide it.

5Public-Facade Parity by Default, Node Runtime as Optional Extension

Axis: Provider behavior coverageConf 78%Cx Medium

Description: Implement the first harness against @swartzrock/byok-runtime only, because the user's six providers are all reachable through listModels and generateText. Add a small architecture note or TODO that a later node-providers command can import @swartzrock/byok-runtime/node for Codex CLI and Claude CLI coverage.

Basis
direct: The public contract test enforces that the main entrypoint exports only the intentional facade, while the README says the node subpath is for local CLI providers (README.md:75, README.md:300). The core provider factory handles the six requested providers directly (src/providers/provider-factory.ts:14).
Rationale
This keeps the harness aligned with consumer usage and avoids accidentally expanding the task into local process execution, command auth, and CLI model-list quirks.
Downsides
Someone reading "all providers" from the enum might expect Codex CLI and Claude CLI too. The README for the fixture should name the scoped provider set explicitly.

6Readiness and Failure UX Without Secret Persistence

Axis: Credential and environment resolutionConf 76%Cx Low

Description: Add a lightweight --check or validation preflight inside each command that reports missing env vars, missing model, or unreachable Ollama before calling generation. Never write API keys, prompts, or responses to disk by default; output goes to stdout/stderr only.

Basis
direct: BYOK's README says the package receives credentials only as call inputs, does not persist or log API keys, and expects trusted host runtimes (README.md:110). Ollama already has URL validation and a default local URL (src/providers/default-deps.ts:4, src/providers/default-deps.ts:12).
Rationale
The harness is going to be used with real credentials. Its most important non-feature is restraint: clear messages, nonzero exits, and no secret storage.
Downsides
Too many custom exit codes can become fake precision. Keep the first version to validation error, provider call failure, and success.

Rejection Summary

# Idea Reason Rejected
1 Published CLI binary Too much product surface for an internal provider smoke harness.
2 Full matrix runner Useful later, but too expensive because it can burn real API quota across all providers accidentally.
3 Auto-pick first model Convenient but risky; first listed model may be deprecated, expensive, or unsuitable.
4 Provider alias soup Small alias support is fine later, but broad aliasing creates ambiguity around claude vs Anthropic vs Claude CLI.
5 JSON-only output Good for automation, worse for quick manual smoke testing.
6 Shell-script wrapper Too brittle for provider-specific option shapes and TypeScript public API checks.
7 Store test profiles on disk Conflicts with BYOK's no-persistence posture and creates secret-handling work.
8 Browser UI tester Nice eventual demo, but the user asked for a CLI-shaped harness.
9 testConnection()-first runtime design Requires the node runtime path when the public facade already covers the requested six providers.
10 Add test harness to published files Increases package surface and support burden without helping local validation.
11 Dry-run mode Useful for debugging arguments, but not enough value for the first CLI because validation errors can show the same resolved provider/key-source summary.
12 Dedicated Ollama status command Mostly duplicates models --provider ollama; keep the Ollama-specific hinting inside the shared command instead.
